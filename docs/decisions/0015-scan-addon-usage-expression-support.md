# 0015. scan-addon-usage: expression usage & the resolution layer

- **Status:** Accepted
- **Date:** 2026-07-24
- **Issue:** [#123](https://github.com/GenvidTechnologies/construct3-chef/issues/123)

## Context

`scan-addon-usage` (#110/#124/#125) reports a plugin/behavior addon's object/family **presence** and its event-sheet **ACE call sites** (conditions/actions), plus effect **application sites**. But C3 **expressions** are not structured `(objectClass, kind, id)` nodes — they live as free text inside condition/action **parameter strings** (`parameters: { "first-value": "Account.SessionLength + 5" }`). So expression usage was invisible: it was the sole remaining gap keeping the **#100** c3addon-tooling umbrella open (ADR [0012](0012-scan-addon-usage-effect-support.md) explicitly deferred it), and #110's "expression-usage mode" was never delivered.

Resolution is **exact, not fuzzy**: every object type/family carries exactly one `plugin-id`, so `Object.expr` resolves to exactly one plugin's expression, and `Object.Behavior.expr` resolves through the object's attached behavior — on par with condition/action matching.

The blocking primitive — a C3 expression tokenizer — shipped upstream in `@genvidtech/c3source@1.8.0` as `extractExpressionReferences(expr): ExpressionToken[]` (a never-throw, string-literal-aware, nesting-aware tokenizer), sequenced behind #136 (which adopted the same 1.8.0 `.c3addon` domain layer and left the `readAddonAcesModel` seam this work consumes). So the local work is **resolution + enumeration + rendering only** — no local tokenizer (the ADR [0006](0006-upstream-ownership-boundary-and-adoption-posture.md) posture: domain parsing upstream, operations/rendering local).

## Decision

**1. A parallel `expressionSites: ExpressionSite[]` collection — not an extension of `CallSite.kind`.** An expression reference has a materially different shape from a `CallSite`: it has **no own `sid`** (it's text inside a parameter, not a node), and it's located positionally by its enclosing node's `jsonPath` + the **parameter key** + a **character span**; its `objectName` is the *referenced* object, not the node's own `objectClass`. This is the same shape-divergence the effect path resolved with a dedicated `effectSites` collection. `ExpressionSite` carries `{sheet, eventNumber, jsonPath, paramKey, objectName, behaviorName?, id, memberName, start, end}`.

**2. Resolution rides the existing `UsageMatcher` seam via a new `matchExpression(token)` method — not a separate `scanExpressionUsage` path.** Unlike effects (ADR 0012), which *cannot* ride the event walk (no ACEs, no event nodes) and so needed `scanEffectUsage`, expressions live **inside the parameter strings the walk already visits**, and they attach to the *same* presence/attribution model as conditions/actions. So the seam fits: `createPluginUsageMatcher` resolves `Object.expr` on a presence object; `createBehaviorUsageMatcher` resolves `Object.Behavior.expr` with the **same union-then-attribution rule** its `matches` uses (instance name in the addon-wide set + the referenced object attributable), so a family-member expression (`Leaderboard.Track.TrackedTime`) attributes to the **family** presence row while the `ExpressionSite.objectName` stays the member (`Leaderboard`). Both factory functions gained an **optional** `AcesModel` parameter (empty-model default) — required to avoid breaking the eight existing 3-arg `createBehaviorUsageMatcher` test call sites; `createPluginUsageMatcher` is now exported for direct testing.

**3. A distinct `expressionSiteCount` on `PresenceRow` — not folded into `callSiteCount`.** An expression reference creates **no new presence row** (a plugin expression requires its object to already be a presence host; a behavior expression requires an attributable host), so only counts change. Folding expression references into `callSiteCount` would silently redefine an existing, rendered field ("condition/action calls"). The row renders `, N expression ref(s)` when non-zero, and a host with **neither** a call site nor an expression reference is the only one that still reads `(instantiated, no ACE calls)`.

**4. Resolution via `AcesModel` + `findExpression`, id-aligned with the diff buckets.** `matchExpression` resolves a token's PascalCase `memberName` through c3source `findExpression(model, expressionName): AceExpression`, taking the `AceExpression.id` (the dash-cased aces.json `id`). The model is built once per scan via `readAddonAcesModel(target)` (the never-throw #136 seam). That `id` is the **same** aces.json `id` that `diffAddonAces` keys its `expression:<id>` changed/removed buckets on, so blast flagging aligns across the resolution path and the diff path with no schema coupling.

## Compromise

- **`CallSite.kind` extension (rejected).** Would have made `sid` nullable and bolted optional `paramKey`/`start`/`end`/`behaviorName`/`memberName` onto `CallSite` — every one meaningless for a condition/action — and the blast/render code keyed on `sid` would special-case the expression variant anyway, so the "unified type" saving is illusory. The effect precedent already settled this shape-split.
- **A separate `scanExpressionUsage` path (rejected).** Would re-walk every event sheet and re-read every objectType/family for data that arrives in the visit the call-site walk already performs. The seam fits because expressions, unlike effects, ride the walk.
- **Blast for a *removed* expression required widening the resolution model.** Acceptance criterion: `--from` must surface a **dangling reference to a removed expression** (the stale-reimport fallout the tool exists to catch). But `matchExpression` resolves against the *current* model, where a removed expression is gone. So in blast mode the model is widened with the diff's removed expressions (reconstructed by their `expressionName` = `AceEntry.scriptName`) — the expression analogue of the existing removed-*key* match-set widening. Without it, `Account.Rank()` after `Rank` was removed would never surface.
- **Known limitation (inherited from #136, recorded in ADR 0014).** `readAddonAcesModel` reads `aces.json` top-level and does not fall through an incomplete extracted dir to the zip; a package with an extracted dir missing `aces.json` yields an empty model (never-throw). Acceptable — the resolution model degrades to "no expressions resolve", never a crash.

## Consequences

- **#100 closes.** Expression usage was its sole remaining child; with this shipped, `scan-addon-usage` covers plugin/behavior condition, action, **and** expression usage plus effect application, so the umbrella can be closed.
- No public-API change beyond `src/c3/addonAceUsage.ts` (already off the `src/index.ts` barrel): the new `ExpressionSite` type / `expressionSites` / `expressionSiteCount` fields are additive and optional, so the ~9 synthetic test result literals compile unchanged; the exported `createPluginUsageMatcher` is off-barrel like its sibling.
- CLI/MCP surfaces are unchanged in signature — expression output flows through the shared `formatAddonUsage`, so it stays byte-identical across both; the MCP tool keeps its `txId` footer (it reads tx-tracked source dirs).
- Fixture coverage lives in `test/fixtures/addon-ace-usage/` (a plugin `SessionLength` CHANGED / `Rank` REMOVED expression pair + a new `GTrack` behavior addon), deliberately **not** the golden `construct3-chef-sample` — no golden churn. Adding it reconciled three pre-existing blast assertions and one `projectObjects` deep-equal (fixture-coupling, expected).
