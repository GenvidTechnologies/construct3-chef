# Plan: Recipe Addressing Redesign

> _Archived session plan from construct3-chef's monorepo development. Historical record; paths map to this repo per the [initiative repository note](../initiative.md) (`bin/`→`src/`). See [archive index](README.md)._

## Branch

Continue on `BUR-0000-recipe-tooling-and-docs`

## Dependencies

Requires `extracted/sid-registry.txt` to exist (generated by the SID generation task, already done).

## Summary

Replace position-based addressing (`events[N]`, `index: N`) with SID-based addressing (`"sid:X"` refs
and `"$symbol"` for new elements) in the recipe system. Fixes the `remove-action` wrong-index bug
and the `autoAdjust` mixed-op-type misplacement bug by making event identification independent of
array position shifts. Deprecates `autoAdjust`.

## Root Cause Recap

Both bugs share the same design flaw: position-based paths (`events[N]`) go stale when any
preceding op inserts or removes events from the same array. `autoAdjust` tries to compensate via
offset arithmetic but only works for homogeneous op types, silently breaking on mixed types.

The fix: identify events by their stable SID instead of their position. Shifts become impossible
to cause problems because SIDs are immutable during recipe execution.

## Design

### New addressing syntax

**Existing elements** — address by SID via the `in` field:
```json
{ "op": "insert-actions", "in": "sid:100234567890123", "after": 0, "actions": [...] }
{ "op": "remove-action", "in": "sid:100234567890123", "index": 2 }
{ "op": "remove-event",  "in": "sid:100234567890456" }
```

**New elements** — assign `id: "$symbol"` at creation, reference via `in: "$symbol"` later:
```json
{ "op": "insert-event", "id": "$loginBlock", "after": "sid:100234567890456", "block": { ... } }
{ "op": "insert-actions", "in": "$loginBlock", "after": 0, "actions": [...] }
```

**Backward compat** — `path` still works for old recipes. `in` takes precedence when both are present.

### What changes in `eventSheetMutator.ts`

New export: `buildSidIndex(sheet: EventSheet): SidIndex` — walks the event tree and builds
`Map<number, SidIndexEntry>` where each entry has `{ node, parentArray, indexInParent }`.
Used by the interpreter to resolve `"sid:X"` refs to event nodes without path arithmetic.

Scope: events only (block, function-block, custom-ace-block, group, variable, comment, include).
Conditions and actions within events are still addressed by `index: N` relative to their stable
parent (once the parent event is addressed by SID, the relative index can't shift).

### What changes in `recipeInterpreter.ts`

`executeFileOps` builds a `SidIndex` at the start of execution for each file, then:
1. For each op, resolves `in` first (if present), falls back to `path`
2. `"sid:XXXXXXX"` → parse → SID index lookup
3. `"$symbol"` → symbol table lookup → SID index lookup
4. When `insert-event` has `id: "$symbol"`, records the new event's SID in the symbol table
5. `after: "sid:X"` on `insert-event` → look up SID in index → derive insertion position

`autoAdjust` is deprecated: the logic is removed, the field is retained with a deprecation
warning. `sortRemoveEventOps` is also removed (it was a workaround for the same problem class;
SID refs don't shift, and for backward-compat position-based recipes, authors should use
descending removes manually or switch to SID addressing).

### What changes in DSL index

`DslIndexEntry` gains `sid?: number`. `formatIndex` adds a SID column (§XXXXXXXXXXXXXXX format,
aligned). Now recipe authors can read the idx, copy the SID, and write `"in": "sid:X"` recipes
without counting positions.

### Named-element shorthand (future)

Functions, groups, and ACEs can be addressed by name (e.g., `"in": "fn:myLoader"`) — out of
scope for this plan but documented in recipe-reference.md as upcoming.

## Risks / Considerations

- **`sortRemoveEventOps` removal**: Existing position-based recipes with consecutive `remove-event`
  ops relied on the implicit sort. After removal, authors must either put removes in descending
  order manually, or switch to SID addressing. Mitigation: emit a warning in `executeFileOps` when
  it detects consecutive `remove-event` ops with position-based paths that would need sorting.
  Alternative: keep `sortRemoveEventOps` as a safety net for position-based recipes only.

- **DSL idx churn**: Adding the SID column changes every idx file. The commit will have a large
  diff (200+ files). This is expected — generated artifact, CI-validated.

- **SID column in action-level rows**: Action-level rows in the idx don't need SIDs (actions are
  addressed by `index` within their parent event). Leave SID blank/empty for action rows.

- **Symbol table scope**: Symbols are per-file, per-recipe execution. No cross-file symbol refs.

- **`after: "sid:X"` on insert-event requires SID to be in the same container**: If the referenced
  SID is in a different parent than the container being inserted into, throw a clear error.
