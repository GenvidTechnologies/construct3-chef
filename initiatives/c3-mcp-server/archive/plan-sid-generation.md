# Plan: Proper SID Generation for Recipe-Generated C3 JSON

> _Archived session plan from construct3-chef's monorepo development. Historical record; paths map to this repo per the [initiative repository note](../initiative.md) (`bin/`→`src/`). See [archive index](README.md)._

## Branch

Continue on `BUR-0000-recipe-tooling-and-docs`

## Dependencies

None — self-contained fix to existing recipe infrastructure.

## Summary

All recipe-generated C3 JSON objects currently use `sid: 0` as a placeholder. C3 is sensitive to
`sid: 0` on action and function parameter objects — not just duplicate non-zero SIDs, but `0`
itself. This plan extracts a shared `sidUtils.ts` module with proper unique SID generation, then
threads it through all builders in `eventSheetMutator.ts` and `instVarMutator.ts`, initialized
per-file from existing SIDs in the recipe interpreter.

## Design

### Approach: SID registry in `extracted/sid-registry.txt` + module-level context

Instead of scanning project files at recipe runtime, maintain a persistent SID registry as part
of the `extracted/` generated artifacts. It fits the existing pattern perfectly:
- Generated by a new 6th generator wired into `npm run generate-c3` / `generateAll.ts`
- Committed to `extracted/` and validated by CI (drift detection catches staleness)
- Regenerated automatically after every `apply-recipe` run (same as DSL/layout summaries)
- Read instantly at recipe start (one file, not 50–80 JSON files)
- Future value: enables SID-based event addressing (like construct3-mcp) as a lookup table

**File format** — `extracted/sid-registry.txt`, tab-separated, one entry per line:
```
100123456789012	eventSheets/Login/LoginEvents.json	sheet
100234567890123	eventSheets/Login/LoginEvents.json	Event1
100345678901234	eventSheets/Login/LoginEvents.json	Event1.conditions[0]
100456789012345	objectTypes/Heroes/HeroData.json	instVar.heroId
```

Columns: `sid TAB source-file TAB human-readable-location`. Sorted by SID for stable diffs.

**Generator scope**: eventSheets (sheet SID, event SIDs, condition SIDs, action SIDs, function
parameter SIDs) + objectTypes (object SID, instVar SIDs). Layouts are out of scope (layout
mutation SIDs are managed by the existing injected `sidGenerator` in `layoutMutator.ts`).

**`bin/c3/sidUtils.ts`** — reads registry, provides module-level context:

```typescript
// bin/c3/sidUtils.ts

let _usedSids: Set<number> | null = null;

/** Initialize from sid-registry.txt. Called once at recipe start. */
export function initSidContext(registryPath: string): void {
  _usedSids = loadSidRegistry(registryPath);
}

/** Initialize directly from a Set (for tests). */
export function initSidContextFromSet(existingSids: Set<number>): void {
  _usedSids = new Set(existingSids);
}

/** Reset context after recipe completes. */
export function resetSidContext(): void { _usedSids = null; }

/** Generate a unique random SID, guaranteed distinct from all existing and generated ones. */
export function generateUniqueSid(): number {
  if (!_usedSids) throw new Error("SID context not initialized — call initSidContext() first");
  let sid: number;
  let attempts = 0;
  do {
    sid = Math.floor(Math.random() * 9e14) + 1e14;  // [1e14, 1e15) — never produces 0
    if (++attempts > 100) throw new Error("SID generation failed after 100 attempts");
  } while (_usedSids.has(sid));
  _usedSids.add(sid);
  return sid;
}

/** Recursively collect all `sid` values from any C3 JSON value (used by the generator). */
export function collectSids(json: unknown): Set<number> { ... }
```

**Why registry over runtime scan:**
- No I/O overhead at recipe start — the registry is a flat text file, not 50–80 JSON parses
- CI enforces it's always current (same drift detection as DSL/layout summaries)
- Falls back gracefully: if registry missing, throw a clear error ("run `generate-c3` first")
- Adds value beyond uniqueness — human-readable SID → location index for future tooling

**Why module-level over parameter threading:**
- No function signature changes to 12 builder functions
- `initSidContextFromSet` enables clean test isolation without file I/O
- `null` guard catches uninitialized use early

**Why a shared `sidUtils.ts`:**
- Avoids `instVarMutator.ts` importing from `eventSheetMutator.ts` (wrong dependency direction)

### Recipe interpreter initialization

`recipeInterpreter.ts` calls `initSidContext("extracted/sid-registry.txt")` **once at recipe
start**, then `resetSidContext()` after all mutations. The generator also writes new SIDs into
the registry during regeneration, so subsequent runs start with an accurate baseline.

## Risks / Considerations

- **Registry missing on fresh clone**: If `extracted/sid-registry.txt` doesn't exist, the
  interpreter throws a clear error ("run `npm run generate-c3` first"). Same behavior as other
  `extracted/` files — already expected by the workflow.
- **Module-level state in tests**: Tests use `initSidContextFromSet()` (no file I/O) and
  `resetSidContext()` in `beforeEach`. Document in test file.
- **`buildLayer` deferred**: `layoutMutator.ts:59` `buildLayer()` stays `sid: 0` — layout layers
  are not action/function parameters. Revisit only if C3 shows sensitivity.
- **Registry includes new SIDs after recipe run**: The recipe interpreter adds generated SIDs to
  the in-memory Set during the run. The post-run regeneration of `extracted/` (including registry)
  picks them up, so subsequent runs start with an accurate baseline.
